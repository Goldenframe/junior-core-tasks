**Описание проекта**
На данный момент проект представляет из себя простой список карточек сотрудников, информация о которых берется с retools. Для запросов используется axios, а данные управляются и берутся через TanStack Query. У каждого компонента свой модуль стилей, в хедере есть селект для управления размером шрифта всей страницы (через rem). Добавление сотрудника происходит через форму, которая появляется с небольшой анимацией через translate, также при монтировании компонента автоматически ставится фокус на первом инпуте засчет ref.current.focus()
**Какие темы рассмотрены в рамках данного мини-проекта:**

    БЭМ / CSS Modules

    Flex vs Grid

    rem / em

    Text ellipsis

    Transform & animation

    Refs

    TanStack Query

    Optimistic update

**1. БЭМ / CSS Modules**

БЭМ и CSS Modules предотвращают конфликтов классов. Но БЭМ — это простометодология именования классов, которая предлагает следующую структуру: блок (компонент), элемент (составная часть блока) и модификатор (состояние или вариация).

CSS Modules уже автоматически генерирует уникальные имена классов. Когда мы импортируем styles из './index.module.css' и прописываем кдасс className={styles.card}, в итоге мы получаем измененный класс в DOM, за счет этого даже если в двух разных компонентах будет эелмент с классом card, эти стили не будут конфликтовать, так как DOM будет принимать их за разные классы. 


**2. Flex vs Grid**

Flexbox и Grid используются для раскладки элементов. Grid представляет двумерную систему, позволяющую управлять позиционированием одновременно по горизонтальной и вертикальной осям. Flexbox в свою очередь реализует одномерную раскладку, то есть позволяет располагать элементы только в одном направлении: либо по вертикали, либо по горизонтали.

Grid применяется для построения сеток с контролем колонок и рядов. Flexbox используется для распределения элементов в одной плоскости.
В рамках выполненного проекта Grid использовался для расположения карточек, а Flexbox для расположения кнопок в header

**3. rem / em**

rem всегда отталкивается от размера шрифта корневого элемента html. Если в корне задано font-size: 16px, то 1rem всегда будет равен 16 пикселям. 

em, в свою  очередь, рассчитывается относительно размера шрифта родительского элемента. Если блоку задать font-size: 20px, то 1em внутри этого блока будет равен 20 пикселям. При вложенности em может привести к неожиданным результатам, так как размеры вычисляются многократно.

**4. Text ellipsis**

Чтобы текст обрезался с многоточием необходимо,чтобы элемент имел явно заданную ширину или максимальную ширину, свойство overflow должно быть выставлено в hidden или auto, и должно присутствовать свойство white-space со значением nowrap для запрета переноса строк. Только при соблюдении всех этих условий свойство text-overflow: ellipsis начинает работать и заменяет обрезанный текст многоточием.

Однако внутри flex-контейнера это перестает работать из-за особенности поведения flex элементов. По умолчанию flex элемент не может стать меньше размера своего контента, поэтому даже если явно задать width, white-space и overflow, текст все равно будет расширять родителя, потому что flexbox пытается сохранить видимость всего контента. чтобы решить эту проблему, необходимо установить min-width: 0 для flex-элемента, что переопределяет это поведение и позволяет сжиматься меньше своего содержимого

Для многострочного обрезания используется другой подход с свойством -webkit-line-clamp, где нужно указать количество видимых строк, а также обязательно задать display: -webkit-box и -webkit-box-orient: vertical и  установить overflow: hidden. В отличие от однострочного варианта, здесь не требуется white-space: nowrap, так как перенос строк разрешен. 

**5. Transform & animation**

Изменение CSS-свойств запускает последовательность операций: reflow это перерасчет геометрии элементов, и repaint это перерисовка пикселей. Свойства width, height, top, left при изменении вызывают reflow, так как влияют на позиционирование и размеры других элементов.

Transform и opacity анимируются без запуска reflow, потому что элемент обрабатывается на отдельном слое. Изменение позиции через translate или прозрачности через opacity затрагивает только сам элемент, не влияя на окружающие. Браузер выполняет только композитинг (наложение слоев), что значительно производительнее.

**6. Refs**

Refs - это способ получить прямой доступ к DOM-элементу в React. Используются для управления фокусом, медиа-элементами или интеграции со сторонними библиотеками. В функциональных компонентах используется хук useRef, который создает объект со свойством current. Изменение ref.current не вызывает повторный рендер компонента, что позволяет хранить мутируемые значения без лишних обновлений.

**7. TanStack Query**

Библиотека для управления серверным состоянием, которая берет на себя загрузку, кэширование, фоновое обновление и повторные попытки запросов. В основе лежат два основных хука: useQuery предназначен для получения данных с сервера, автоматически кэширует результаты по уникальному ключу и синхронизирует состояние между компонентами, а useQuery также предоставляет информацию о статусе загрузки и ошибках. useMutation используется для операций изменения данных на сервере и не кэшируется по умолчанию, но позволяет после выполнения запроса обновить связанные кэши через инвалидацию, чтобы данные в приложении оставались актуальными.

Query предназначен для чтения данных с сервера, запросы кэшируются и могут автоматически перезапрашиваться. Mutation используется для создания, обновления или удаления данных и после успеха обычно происходит обновление связанных query (через queryKey)

**8. Optimistic update**

Паттерн, при котором интерфейс обновляется сразу после действия пользователя, не дожидаясь ответа сервера. Если запрос на сервер завершается ошибкой,то изменения откатываются к предыдущему состоянию. Пример работы в TanStack Query: в onMutate сохраняется предыдущее состояние данных и применяются оптимистичные изменения, в onError происходит откат к сохраненному состоянию

Главный риск optimistic update -рассинхронизация с сервером, когда оптимистичные изменения не совпадают с реальным ответом или запрос завершается ошибкой. Также возможны состояния гонки при нескольких последовательных мутациях. 

**9. Derived state (вычисляемые значения)**

Некоторые данные не нужно хранить в state, потому что они уже есть в пропсах или других данных. Если положить их в state и синхронизировать через useEffect, получится лишний код и риск ошибок. Лучше вычислять такие данные прямо при рендере. В проекте компонент EmployeeStatsGood вычисляет количество сотрудников и список должностей сразу из данных useEmployees

**10. Методы массивов (map, filter, reduce)**

Вместо императивных циклов for, в проекте используются декларативные методы массивов. filter применяется для поиска сотрудников по строке запроса, возвращая новый массив с отфильтрованными элементами. map преобразует массив сотрудников в массив должностей. reduce подсчитывает частоту встречаемости каждой должности. В компоненте EmployeeStatsBad показан пример с вложенными циклами, а в EmployeeStatsGood пример с Set и методами массивов

**11. Set и Map**

Set -это структура данных, хранящая только уникальные значения. В проекте Set используется для получения списка уникальных должностей сотрудников: [...new Set(employees.map(emp => emp.job_title))]. Map хранит пары ключ значение, где ключом может быть любой тип данных. Map.get() и Map.set() работают быстрее обычного объекта при частых операциях добавления и удаления. В проекте Map используется для подсчета количества сотрудников по каждой должности.

**12. Замыкания (Closures)**

Замыкание - это способность функции запоминать переменные из внешней функции даже после того, как внешняя функция завершила выполнение. В проекте хук useModalCounter использует замыкание через useRef. Переменная countRef.current сохраняется между рендерами компонента, а функции increment, getCount и reset имеют к ней доступ. Без замыкания пришлось бы хранить счетчик в глобальной переменной или передавать его через props

**13. Контекст this и bind**

Значение this определяется тем, как вызвана функция, а не где она объявлена. При передаче метода объекта как callback, контекст теряется и this становится undefined. В проекте в компоненте EmployeeCard показана эта проблема: если вызвать logger.logDelete как callback, this.name будет undefined. Исправление - метод bind, который создает новую функцию с навсегда привязанным контекстом: const boundLogDelete = logger.logDelete.bind(logger).

**14. Event Loop (очередь событий)**

JavaScript выполняется в одном потоке. Event Loop управляет порядком выполнения кода. Сначала выполняется весь синхронный код (Call Stack). Затем выполняются все микрозадачи (Promise.then, async/await после await, queueMicrotask). Затем выполняется одна макрозадача (setTimeout, setInterval, события DOM). Цикл повторяется. В проекте в MainPage добавлена демонстрация, где console.log выводится в порядке: синхронный код, Promise, продолжение async, setTimeout. 

**15. useTransition**

Когда быстро печатам в поиске, каждое нажатие клавиши вызывает фильтрацию списка. Без useTransition при большом списке сотрудников ввод будет залипать, потому что React занят фильтрацией. С useTransition сначала обрабатывает ввод, а фильтрацию делает в фоне, не блокируя интерфейс.

**16. CSS специфичность (Specificity)**

Браузер применяет то CSS правило, у которого выше специфичность. Специфичность считается так: ID (1,0,0) > класс/псевдокласс/атрибут (0,1,0) > тег (0,0,1). В проекте в компоненте Modal в CSS комментариях показана проблема: селектор по тегу button имеет специфичность 0,0,1 и не может переопределить стиль класса .modal__close с специфичностью 0,1,0. Решение - использовать селектор с классом или увеличить специфичность (добавить родительский класс). !important ломает каскад и его следует избегать.

**17. Memoization (React.memo, useMemo, useCallback)**

Мемоизация - это кэширование результатов вычислений или компонентов для предотвращения лишних рендеров.

React.memo - обертка для компонента, которая предотвращает перерендер, если пропсы не изменились. В проекте EmployeeCard обернут в memo, так как в списке много карточек и без memo они все перерендеривались бы при поиске или добавлении нового сотрудника.

useMemo - хук для кэширования результатов тяжелых вычислений. В BadRerenderDemo heavyCalculation обернут в useMemo, чтобы не выполняться при каждом рендере.

useCallback - хук для кэширования функций, чтобы они не пересоздавались при каждом рендере. В BadRerenderDemo handleUserClick обернут в useCallback.

Мемоизация не нужна везде: она усложняет код, потребляет память и бесполезна для легковесных вычислений или компонентов, которые и так редко рендерятся.

**18. DOM Events (Делегирование)**

Делегирование - это техника, когда обработчик события вешается на родителя, а не на каждого ребенка. Благодаря всплытию (bubbling) клик по ребенку достигает родителя.

В проекте в EmployeeList один обработчик onClick на div.list обрабатывает клики по всем карточкам. Метод closest() определяет, по какому элементу кликнули (кнопка удаления или сама карточка).

**19. vh / dvh / svh**

На мобильных устройствах 100vh включает адресную строку браузера, что приводит к обрезанию контента или лишней прокрутке.

    svh (small viewport height) - минимальная высота, когда адресная строка показана.

    lvh (large viewport height) - максимальная высота, когда адресная строка скрыта.

    dvh (dynamic viewport height) - динамическая высота, которая меняется при скрытии или появлении адресной строки.

svh и lvh статичны, они не меняются во время сессии пользователя. dvh меняется, но может вызывать скачки контента при скролле. Для container queries используется свойство container-type, которое может принимать значения inline-size (отслеживает ширину), block-size (отслеживает высоту), size (отслеживает и ширину, и высоту), normal (отключает отслеживание). inline-size используется чаще всего, так как обычно важнее ширина контейнера. size реже применяется, так как отслеживание двух осей одновременно сильнее влияет на производительность.

В проекте модальное окно на мобильных устройствах получает height: 100dvh. Для старых браузеров добавлен fallback: height: 100vh.

**20. @media / @container**

@media запросы смотрят на ширину viewport (окна браузера). @container запросы смотрят на ширину родительского контейнера.

В проекте карточка сотрудника использует @container. При ширине карточки меньше 300px кнопка растягивается на всю ширину. При ширине больше 400px карточка становится горизонтальной. Это позволяет карточке адаптироваться к месту, где она находится (например, в узкой колонке или широкой сетке), независимо от ширины экрана.

Container queries удобно применять для переиспользуемых компонентов по типу карточек, виджетов, панелей инструментов, компонент сам решает, как ему сверстаться в зависимости от выделенного ему места. однако не стоитсоздавать много вложенных контейнеров друг в друга, потому что каждый контейнер с container-type создает новый контекст, и браузеру нужно отслеживать изменения размеров каждого из них, а это может негативно повлиять на производительность.

**21. RegExp (Регулярные выражения)**

Регулярные выражения - это шаблоны для поиска и проверки строк. В JavaScript записываются между слешами: /pattern/flags.

Синтаксис:

    ^ - начало строки

    $ - конец строки

    \d - любая цифра

    \w - буква, цифра или подчеркивание

    \s - пробел

    [abc] - любой из перечисленных символов

    [a-z] - диапазон символов

    {n,m} - от n до m повторений

    ? - 0 или 1 раз

    + - 1 или более раз

    * - 0 или более раз

    (x|y) - или x, или y

Флаги:

    i - регистронезависимый поиск

    g - глобальный поиск

    m - многострочный режим

Примеры:

    /^\d{3}-\d{2}-\d{4}$/ - номер вида 123-45-6789

    /^[a-zA-Z\s]{2,}$/ - только буквы и пробелы, минимум 2 символа

    /^[\w.-]+@([\w-]+\.)+[a-z]{2,}$/i - email

Методы:

    test() - проверяет, соответствует ли строка шаблону, возвращает true или false

    match() - ищет совпадения и возвращает массив

    replace() - заменяет найденное

**22. Context API**

Context API нужен, чтобы передавать данные глубоко в компоненты без пробрасывания пропсов через каждый уровень.

Сначала создается контекст через createContext(). Затем компонент-провайдер оборачивает дочерние компоненты и передает им значение. Внутри любого дочернего компонента это значение можно получить через хук useContext().

В проекте есть FontSizeContext для размера шрифта и ShowModalContext для управления модалкой. В App оба провайдера оборачивают MainPage, а в хедере и других местах значения забираются через кастомные хуки useFontSize и useShowModal.

Также в React 19 появилась функция use(), которая может читать контекст внутри условных операторов и циклов, в отличие от useContext, который работает только на верхнем уровне компонента.

Context не подходит для любых глобальных данных. При каждом изменении значения все компоненты, которые на него подписаны, перерендериваются. Для часто меняющихся данных (например, позиция мыши) лучше использовать другие решения.
